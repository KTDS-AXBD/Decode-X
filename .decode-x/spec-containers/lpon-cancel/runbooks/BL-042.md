# Runbook — BL-042: 결제 망 취소 처리

## 트리거
- 외부 결제망(BC카드 망/VAN)에서 취소 이벤트 수신 → SYSTEM이 `processNetworkCancel(db, paymentId, idempotencyKey)` 호출

## 처리 흐름
1. **Idempotency check**: `SELECT id FROM payments WHERE idempotency_key = ? AND status = 'NETWORK_CANCELED'`
   - 발견 시 → `{cancelId: existing.id, status: 'ALREADY_PROCESSED'}` 즉시 반환 (idempotent response, 외부 망 재시도 안전)
2. **Payment 검증**: `SELECT id, voucher_id, amount, status FROM payments WHERE id = ?`
   - 미존재 또는 `status != 'PAID'` → `CancelError('E409')` throw
3. **Atomic transaction** (`db.transaction(() => { ... }); tx();`):
   - `cancel_transactions` INSERT (cancel_type='NETWORK', requester_type='SYSTEM', status='COMPLETED', reason='Network cancel detected')
   - `payments.status` → `'NETWORK_CANCELED'` + `canceled_at = now`
   - `vouchers.balance += payment.amount` (잔액 복구)
4. **Return**: `{cancelId, status: 'NETWORK_CANCELED'}`

## 모니터링
- Network cancel 이벤트 수신 빈도 (대시보드 metric)
- ALREADY_PROCESSED 응답 비율 — 외부 망 retry rate 추정용
- E409 비율 — 잘못된 paymentId 또는 이미 다른 경로로 취소된 건

## 알림
- ALREADY_PROCESSED rate > 10% 지속 시 → 외부 망 retry 폭주 의심, infra 점검
- E409 spike → Stage 2 결제 처리 흐름 이상 신호

## 트러블슈팅
- **Atomic 실패**: SQLite transaction rollback 자동. cancel_transactions 부분 INSERT 흔적 없음 보장.
- **잔액 음수 우려 없음**: cancel은 balance 증가만 (`+=`), 음수 발생 불가
- **payments status race**: SQLite single-writer 보장 + transaction 내부에서 SELECT-UPDATE 분리 → 동시성 안전
