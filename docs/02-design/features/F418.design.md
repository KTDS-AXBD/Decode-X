# F418 Design — PolicyCandidateSchema exception 필드 정식화

## 1. 배경 및 Root Cause

### 현재 구조의 문제

```typescript
// packages/types/src/policy.ts (line 16-25) — 인퍼런스 결과 schema
export const PolicyCandidateSchema = z.object({
  title: z.string(),
  condition: z.string(),    // When
  criteria: z.string(),     // If (decision logic)
  outcome: z.string(),      // Then
  policyCode: z.string(),
  sourcePageRef: z.string().optional(),
  sourceExcerpt: z.string().optional(),
  tags: z.array(z.string()).default([]),
  // ❌ exception 필드 부재
});

// packages/types/src/skill.ts (line 32-46) — 패키지 내부 schema
export const PolicySchema = z.object({
  code, title, description, condition, criteria, outcome,
  source: { documentId, pageRef?, excerpt? },
  trust, tags,
  // ❌ exception 필드 부재
});

// services/svc-skill/src/ai-ready/spec-content-adapter.ts (line 79) — runbook 렌더
const exception = p.source.excerpt ?? "—";
// ❌ source.excerpt(원문 발췌)를 exception(Else 분기)으로 잘못 매핑
//    → LPON 35건 전부 "—" 출력 → exception_handling rubric 0/35

// services/svc-policy/src/prompts/policy.ts (buildSystemPrompt) — inference prompt
// ❌ "exception" 필드 없음, Else 분기 출력 지시 없음
//    → svc-policy LLM이 Else 분기 추론 자체를 안 함
```

### 4-Layer 결함 chain

```
[L1 prompt] svc-policy inference prompt에 exception 항목 없음
   ↓
[L2 schema] PolicyCandidateSchema에 exception 필드 없음 (LLM JSON 출력 후 zod parse도 통과)
   ↓
[L3 schema] PolicySchema(번들 내부)에 exception 필드 없음 (svc-skill bundler가 옮기지 못함)
   ↓
[L4 runbook] spec-content-adapter가 exception 컬럼에 source.excerpt 잘못 매핑 → "—" 출력
```

F417 augmentation은 **L4 fallback** — description 필드에 LLM augment하여 evaluator 입력만 보강. L1~L3 미수정.

## 2. 설계 결정 (Design Decisions)

### DD-1: Schema 확장 형태
- **Option A** (채택): `exception?: string` optional 단일 필드 — 자연어 Else 분기 설명. 기존 condition/criteria/outcome과 일관.
- Option B: `exception?: { trigger: string; outcome: string }` 구조 — 과도. condition/criteria/outcome도 자연어 단일 필드.
- Option C: `exceptions?: Array<{ ... }>` — multi-exception. 현 단계 over-engineering.

**채택 사유**: 점진 도입 + LLM 출력 안정성 + 기존 필드 일관성.

### DD-2: Backfill 정책 (Plan §4 사용자 결정)
**(c) 둘 다 — D1 + R2 augmented prefix**

| 경로 | 목적 | 구현 |
|------|------|------|
| D1 `db-policy.policies.exception` | SSOT 정렬, 신규 inference 시점부터 자연 채움 | migration `0006_policies_add_exception.sql` (조건부 — 기존 schema 점검 후) + Haiku 1-shot UPDATE 43건 |
| R2 `skill-packages/augmented/{id}.skill.json` | F417 description augmentation 결합, evaluator 입력 최대 보강 | `scripts/policy/backfill-exception.ts`가 D1 UPDATE + augmented bundle rewrite (exception 필드 추가) 동시 수행 |

evaluator 우선순위 (이미 F417에서 구현):
```
1. r2KeyOverride 명시 (PoC/실험) → R2 augmented
2. 기본 호출 → D1 r2_key (F415 fix) → R2 정상 경로
3. fallback → loadSpecContentLegacy (spec-container)
```

### DD-3: svc-policy prompt 갱신 위치
`services/svc-policy/src/prompts/policy.ts:buildSystemPrompt()` 안의 "Each object must have these exact fields:" 블록에 `exception` 항목 추가:

```typescript
// 추가될 prompt 섹션:
- exception (string, optional): "Else" branch description — what happens when the condition is
  NOT met OR the criteria fails. Examples: error responses, alternative paths, fallback policies,
  rejection reasons. Use natural Korean. If no clear Else branch exists in the source, omit this
  field (do not output empty string).

Example with exception:
{
  "title": "주택 구입 중도인출",
  "condition": "가입자가 무주택 세대주이고 본인 명의 주택 구입 자금이 필요한 경우",
  "criteria": "최근 3개월 이내 주택 매매계약서 또는 분양계약서 제출",
  "outcome": "가입금액의 100% 한도 내 1회 인출 승인",
  "exception": "주택 매매계약서가 가족 명의이거나 6개월 이상 경과한 경우 인출 거절 + 사유 통보",
  "policyCode": "POL-PENSION-WD-HOUSING-001",
  ...
}
```

### DD-4: skill-builder runbook 매핑 정정
```diff
// services/svc-skill/src/ai-ready/spec-content-adapter.ts (line 79)
- const exception = p.source.excerpt ?? "—";
+ const exception = p.exception ?? "—";
```

`Policy` 타입에 `exception?: string`이 추가되므로 컴파일 통과. 기존 정책(미backfill)은 fallback "—" 유지(회귀 없음).

점검 결과(`grep -rn "PolicyCandidateSchema\|PolicySchema" ~/work/axbd/Foundry-X` → 0건). Decode-X 단독 schema. Foundry-X repo 변경 없음.

### 3.6 scripts/policy/backfill-exception.ts (신규)
```typescript
// 의사 코드:
// 1. ENV 체크 (OPENROUTER_API_KEY, INTERNAL_API_SECRET, target organization)
// 2. SELECT id, code, condition, criteria, outcome, organization_id
//    FROM policies WHERE skill_id IN (LPON 35 + lpon 8 정책 매핑) AND exception IS NULL
// 3. for each policy (sequential, concurrency=2):
//    a. Haiku call with prompt: "Given condition/criteria/outcome, infer the Else branch (what happens when criteria fails or condition not met). Return single sentence Korean. If unclear, return null."
//    b. response가 null/empty면 skip
//    c. UPDATE policies SET exception = ? WHERE id = ?
// 4. for each affected skill_id:
//    a. R2 augmented bundle 갱신 — F417 augmented bundle JSON load + policies[i].exception 추가 + R2 write back
//    b. 또는 svc-skill /skills/{id}/rebundle 호출 → 정상 경로 R2 갱신
// 5. log summary: total updated, success/fail, total cost
```

추정 비용: 43 calls × Haiku $0.025/Mtok input + $0.125/Mtok output × ~500 tok input + ~50 tok output ≈ **$0.05** (매우 저렴). 안전 cap $2.

### 3.7 scripts/ai-ready/single-eval-loop.ts 재실행
F417에서 신설된 스크립트 그대로 사용. 옵션:
- `--organization LPON --use-augmented` (R2 augmented 우선)
- `--organization LPON` (D1 r2_key 정상 경로 — F418 backfill 효과 직접 측정)

두 모드 비교 측정으로 "augmentation prefix vs schema 정식화" 효과 분리 분석.
