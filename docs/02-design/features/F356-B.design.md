## 2. D1 Schema (`infra/migrations/db-skill/0012_ai_ready_scores.sql`)

### 2.1 `ai_ready_scores` 테이블

```sql
CREATE TABLE ai_ready_scores (
  id TEXT PRIMARY KEY,                        -- uuid v4
  skill_id TEXT NOT NULL,
  organization_id TEXT NOT NULL,              -- 'LPON' | 'Miraeasset' (multi-org 대비)
  criterion TEXT NOT NULL CHECK (criterion IN (
    'source_consistency','comment_doc_alignment','io_structure',
    'exception_handling','srp_reusability','testability'
  )),
  score REAL NOT NULL CHECK (score BETWEEN 0 AND 1),
  rationale TEXT NOT NULL,
  passed INTEGER NOT NULL CHECK (passed IN (0,1)),
  pass_threshold REAL NOT NULL DEFAULT 0.75,
  model TEXT NOT NULL CHECK (model IN ('haiku','opus','sonnet')),
  batch_id TEXT,                              -- NULL for ad-hoc single eval
  evaluated_at TEXT NOT NULL,                 -- ISO 8601 UTC
  cost_usd REAL NOT NULL DEFAULT 0,
  FOREIGN KEY (skill_id) REFERENCES skills(skill_id)
);

CREATE INDEX idx_ai_ready_skill ON ai_ready_scores(skill_id, model, evaluated_at DESC);
CREATE INDEX idx_ai_ready_batch ON ai_ready_scores(batch_id) WHERE batch_id IS NOT NULL;
CREATE INDEX idx_ai_ready_org   ON ai_ready_scores(organization_id, evaluated_at DESC);
```

### 2.2 `ai_ready_batches` 테이블

```sql
CREATE TABLE ai_ready_batches (
  batch_id TEXT PRIMARY KEY,                  -- uuid v4
  organization_id TEXT NOT NULL,
  model TEXT NOT NULL,
  parent_batch_id TEXT,                       -- Opus cross-check가 Haiku batch를 참조
  status TEXT NOT NULL CHECK (status IN ('queued','running','completed','failed','partial')),
  total_skills INTEGER NOT NULL,
  completed_skills INTEGER NOT NULL DEFAULT 0,
  failed_skills INTEGER NOT NULL DEFAULT 0,
  total_cost_usd REAL NOT NULL DEFAULT 0,
  started_at TEXT NOT NULL,
  completed_at TEXT,
  metadata_json TEXT                          -- sampling mode, retry count 등
);

CREATE INDEX idx_ai_ready_batches_org ON ai_ready_batches(organization_id, started_at DESC);
```

### 2.3 데이터 수명 주기

- 단건 재평가는 새로운 row INSERT(이력 보존). 조회 시 `model + evaluated_at DESC` 최신순 1건 반환.
- 배치 완료 후 `ai_ready_batches.status = 'completed'`. 중간 실패 skill은 `failed_skills` 증가 + DLQ row 기록.
- 5,154 row × 평균 200 bytes ≈ 1MB. D1 500MB 한도에서 여유.

## 3. API Design

### 3.1 `POST /skills/:id/ai-ready/evaluate` (단건 동기)

**Request**
```json
{
  "model": "haiku",          // 필수. haiku | opus | sonnet
  "force": false             // default false. 최근 평가 < 1h면 cached 반환
}
```

**Response (200)**
```json
{
  "skillId": "…",
  "model": "haiku",
  "criteria": [ { "criterion": "source_consistency", "score": 0.92, "rationale": "…", "passed": true, "passThreshold": 0.75 }, … 6종 ],
  "totalScore": 0.734,
  "passCount": 4,
  "overallPassed": true,
  "evaluatedAt": "2026-04-24T…Z",
  "costUsd": 0.004
}
```

**Errors**: 401 (unauthorized), 403 (RBAC), 404 (skill not found), 409 (batch in progress for this org+model), 429 (daily cost guard), 500/502/UPSTREAM_ERROR.

**RBAC**: Analyst + Developer + Reviewer. Client 차단.

### 3.2 `POST /skills/ai-ready/batch` (비동기 트리거)

**Request**
```json
{
  "model": "haiku",                       // 필수
  "organizationId": "LPON",
  "crossCheckSampleSize": 100,            // optional. Haiku batch 완료 후 자동 Opus cross-check
  "dryRun": false                         // default false. true면 skill list만 반환
}
```

**Response (202)**
```json
{
  "batchId": "…",
  "totalSkills": 859,
  "estimatedCostUsd": 19.88,
  "estimatedDurationMinutes": 34,
  "crossCheckBatchId": null,              // 자식 batch는 Haiku 완료 후 생성
  "status": "queued"
}
```

**Errors**: 400(invalid model/org), 409 (existing batch running for same org+model), 429 (cost guard, expected daily > $50).

**RBAC**: Developer + Reviewer. Analyst 차단(비용 통제).

### 3.3 `GET /skills/:id/ai-ready/evaluations`

**Query**: `?model=haiku&criterion=source_consistency&limit=20&cursor=…`

**Response (200)**: `{ evaluations: [...], nextCursor: "..." }`

### 3.4 `GET /skills/ai-ready/batches/:batchId`

**Response (200)**
```json
{
  "batchId": "…",
  "status": "running",                    // queued|running|completed|failed|partial
  "totalSkills": 859,
  "completedSkills": 327,
  "failedSkills": 2,
  "progressPct": 38.1,
  "totalCostUsd": 7.56,
  "startedAt": "2026-04-24T…Z",
  "completedAt": null,
  "childBatchId": null,
  "avgScore": 0.731                       // 완료 skill 기준 이동 평균
}
```

## 4. Queue Orchestration

### 4.1 `wrangler.toml` 설정 (svc-skill)

```toml
[[queues.producers]]
queue = "ai-ready-queue"
binding = "AI_READY_QUEUE"

[[queues.consumers]]
queue = "ai-ready-queue"
max_batch_size = 1                  # 1 message = 1 skill
max_batch_timeout = 1
max_concurrency = 10                # 30분 목표 달성 핵심 설정
max_retries = 3
dead_letter_queue = "ai-ready-dlq"

[[queues.producers]]
queue = "ai-ready-dlq"
binding = "AI_READY_DLQ"
```

### 4.2 Consumer 처리 흐름 (`queue/ai-ready-consumer.ts`)

```ts
export async function handleAIReadyMessage(
  msg: Message<{ batchId: string; skillId: string; model: string; organizationId: string }>,
  env: Env,
  ctx: ExecutionContext,
) {
  const { batchId, skillId, model } = msg.body;
  try {
    const pkg = await loadSkillPackage(env, skillId, msg.body.organizationId);
    const spec = buildSpecContent(pkg);
    const evaluation = await runSixCriteriaEvaluation(env, spec, model);
    await insertScores(env, batchId, skillId, evaluation);
    await updateBatchProgress(env, batchId, { completed: 1, costUsd: evaluation.costUsd });
    msg.ack();
  } catch (e) {
    if (msg.attempts >= 3) {
      await updateBatchProgress(env, batchId, { failed: 1 });
      await recordDlq(env, batchId, skillId, String(e));
      msg.ack();
    } else {
      msg.retry({ delaySeconds: 5 * msg.attempts });
    }
  }
}
```

### 4.3 Cross-check Auto-trigger

Haiku batch `completed` 시점에 consumer 마지막 message 핸들러가 `crossCheckSampleSize > 0`이면 자동으로 Opus batch를 생성:
1. 완료된 skill 중 random 100건 sampling
2. 신규 batchId 발급 + `parent_batch_id` 설정
3. Opus message fan-out (동일 Queue, `model: 'opus'` 메타)

## 5. Evaluator Module (`ai-ready/evaluator.ts`)

F356-A의 `scripts/ai-ready/evaluate.ts` 핵심 로직 이식. 주요 차이:

| 항목 | F356-A (CLI) | F356-B (Worker) |
|------|--------------|-----------------|
| Runtime | Node.js | Cloudflare Worker (V8) |
| LLM 호출 | `callOpenRouterWithMeta` 직접 | `callLlmRouterWithMeta` (env.ts 통합, TD-44 결과) |
| 파일 I/O | `fs/promises` | R2 get + D1 (fs 없음) |
| 비용 가드 | 실시간 `/usage` 폴링 | batch 레벨 `ai_ready_batches.total_cost_usd` 체크 |
| JSON 파싱 | `JSON.parse` + zod | 동일 (Zod schema 공유) |
| 동시성 | Promise.all chunks | Queue consumer concurrency 10 |

공유 모듈: `ai-ready/prompts.ts`(rubric v2, F356-A 검증 완료)는 그대로 import.

## 6. Batch CLI (`scripts/ai-ready/batch-evaluate.ts`)

### 6.1 Usage

```bash
# Production (실제)
pnpm tsx scripts/ai-ready/batch-evaluate.ts \
  --env production \
  --model haiku \
  --organization LPON \
  --cross-check 100

# Dry-run (10건 샘플 + 비용 추정만)
pnpm tsx scripts/ai-ready/batch-evaluate.ts \
  --env production --model haiku --dry-run
```

### 6.2 흐름

1. svc-skill API `POST /skills/ai-ready/batch` 호출 → `batchId` 획득
2. 5초 간격 `GET /skills/ai-ready/batches/:batchId` 폴링 (최대 45분)
3. `status=completed` 도달 시:
   - Haiku raw 5,154 점수 retrieve → `reports/ai-ready-full-{date}.json` 저장
   - cross-check batch도 동일 방식으로 대기 + 병합
4. KPI 집계 → `reports/ai-ready-full-{date}.md` 마크다운 작성(Node.js에서 계산)

### 6.3 비용 가드

- 사전 추정: `estimatedCostUsd` > $50 시 AskUserQuestion(계속/중단/sampling)
- 실시간: 폴링 응답의 `totalCostUsd` > $45 시 경고 출력(중단은 API 레이어 책임)

## 9. Observability

### 9.1 Structured Logs

- `createLogger('svc-skill:ai-ready')` 사용
- Key fields: `batchId, skillId, criterion, model, score, costUsd, durationMs, attempts`
- 실패 시 `logger.error({ batchId, skillId, error, stack })`

### 9.2 Metrics

- `ai_ready_batches.total_cost_usd` = 실시간 cost 집계
- `ai_ready_scores` 분포는 CLI `ai-ready-full-{date}.md` 리포트에서 정적 집계

### 9.3 Error Handling

- LLM timeout → Queue retry 3회
- JSON parse 실패 → retry 1회 후 DLQ (msg 메타에 raw response 기록)
- D1 INSERT 실패 → Queue retry (idempotent: `INSERT OR IGNORE` with UUID key)
- R2 load 실패 → failed_skills 증가, DLQ 기록

## 10. Migration Plan

### 10.1 D1 Apply

1. Local: `npx wrangler d1 execute db-skill --local --file=infra/migrations/db-skill/0012_ai_ready_scores.sql`
2. Production: CI (`deploy-services.yml`) `migrate-d1` job이 자동 적용 — TD-39 해소 이후 자동 파이프라인 작동 증명됨 (세션 223)
3. `d1_migrations` 테이블에 row 자동 기록

### 10.2 Worker Deploy

1. `wrangler.toml` queue 설정 추가 → deploy 시 CF가 Queue binding + DLQ 자동 생성
2. 기존 라우트 영향 없음 (신규 path만 추가)
3. 기존 테스트 regression 없음 (TD-45 해소로 114 test files PASS 기준선)

### 10.3 Rollback

- `0012` migration은 idempotent(CREATE TABLE IF NOT EXISTS 패턴)
- Queue 설정 롤백 시 queue는 유지(CF는 drain 후 삭제)
- Worker 롤백 시 D1 테이블만 잔존(cost 영향 0)
