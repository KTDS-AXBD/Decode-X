## Part 1: 왜 (목적/문제)

**Q: 이 스펙을 왜 정의해야 하나요?**

A: 세 가지 문제가 있음:

1. **추출물의 상세도 부족**: AI Foundry가 역공학으로 추출한 결과물(policies, ontologies, skills)을 사람이 보고 맞고 틀리고를 판단할 수 있는 수준이어야 함. 예: "온누리상품권의 결제 취소는 어떤 경우에 어떻게 처리해야 한다"는 구체적 시나리오가 필요.

2. **AI Agent 구현 가능성 판단 기준 부재**: 주어진 스펙만으로 AI Agent가 충분히 결과물을 만들 수 있는 수준인지 판단할 기준이 없음. 아키텍처 구성, 데이터 구조, 중요 로직의 정책/조건 수용 등 세밀한 정의 필요.

3. **최소 개입 Working Version 조건 미정의**: 최소한의 사람 개입으로 Working Version(웹/앱 사이트)이 구현될 수 있는 조건이 정의되어 있지 않음.

**Q: 시급성은?**

A: 의사결정권자(본부장)가 역공학 결과물의 스펙 수준을 직접 확인하고 싶어함. Foundry-X 제품화의 선결 조건은 아니지만, 본부장에게 가시적 산출물을 보여줘야 하는 상황. 바로 지금 필요.

## Part 2: 누구를 위해 (사용자)

**Q: 주 사용자는?**

A:
- **1차 사용자 — 본부장**: 스펙의 품질/수준을 보고 판단. Go/No-Go 결정권자.
- **2차 사용자 — AI Agent + SI 개발자**: 스펙을 입력으로 받아 실제 시스템을 구현하는 주체. AI Agent(Claude Code 등)가 VibeCoding으로 자동 생성하고, SI 개발자가 보완.

**Q: 스펙 형식에 대한 기대?**

A: 새로운 형식 OK. 기존 SI 산출물 형식(요구사항 정의서, 화면 설계서 등)에 얽매일 필요 없음. 품질과 수준이 확인되면 형식은 상관없음.

## Part 3: 무엇을 (범위)

**Q: 핵심 기능 한 문장?**

A: "AI Foundry의 역공학 결과물(policies, ontologies, skills)을 Claude Code가 바로 시스템을 만들 수 있는 수준의 개발 스펙으로 재구성하는 것"

**Q: 구성 요소 우선순위?**

A:
1. **비즈니스 로직** (최우선) — 정책, 조건, 시나리오
2. **데이터 구조** — 테이블, 관계, 제약조건
3. **기능 정의서** — 기능 목록, 입출력, 플로우
4. **아키텍처 정의서** — 시스템 구성, 레이어, 통신 방식
5. **API 명세** — 엔드포인트, 요청/응답
6. **화면 설계** (후순위)

**Q: 입력 소스?**

A: 3가지 소스 모두 활용:
- AI Foundry 추출물 (policies, ontologies, skills)
- 원본 SI 산출물 (요구사항 정의서, 테이블 정의서, 소스코드 등)
- 외부 도메인 지식 (법규, 업계 표준 등)

## Part 4: 성공 기준

**Q: 성공 기준?**

A: 두 가지 모두 충족:
1. **본부장 승인**: 스펙의 품질/수준을 보고 Go 판정
2. **Working Version 생성 가능**: 스펙만으로 AI Agent가 동작하는 시스템을 만들어낼 수 있음

**Q: MVP 수준?**

A: "그 사이" — 핵심 비즈니스 로직은 실제 동작하는 수준, 부가 기능은 목업/스텁 수준.
