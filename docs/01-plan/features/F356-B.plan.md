## 1. Overview

### 1.1 Purpose

F356-A에서 확정한 **6기준 rubric(소스 정합성/주석 문서/IO 구조/예외 핸들링/SRP/테스트 가능성)** 및 **rubric v2 프롬프트**를 LPON 전수(859 skill × 6기준 = **5,154 점수**)에 적용하여, AI-Ready 품질 분포 전체를 확보한다. 또한 실 운영 API 엔드포인트로 노출하여 본부장 리뷰 및 외부 시연에서 즉시 재평가 가능한 상태를 구축한다.

### 1.2 Background

- **선행 해소**: F356-A(Sprint 230 MERGED) + TD-42(코드) + TD-43(실 실행 + rubric v2 정확도 100%) + TD-44/45(OpenRouter via CF AI Gateway 안정화)
- **Phase 2 완전 조건**: F356-A Out-of-Scope 4건 (API endpoint + D1 migration + 전수 배치 + Opus/Haiku 교차검증)을 본 Sprint에서 모두 처리
- **PRD §C.1 확정**: 6기준 rubric + passThreshold 0.75 + 4기준 이상 통과 시 AI-Ready PASS

### 1.3 Scale 산정

- **LPON 전수**: 859 skill (SPEC §5 Production Data 기준)
- **점수**: 859 × 6 criteria = **5,154 점수**
- **Haiku 전수 비용** ($~18): 세션 234 실측 42 call $0.162 → 5,154 call $19.88
- **Opus 교차검증 비용** ($~30): 100 skill × 6 = 600 call, Haiku 대비 ~20x 단가 가정 → $~28
- **예상 실행 시간** (concurrency 10, 호출당 4s): 5,154 / 10 × 4s = 2,062s ≈ **34분** (30분 목표 근접)

## 3. Execution Plan — 4 Step × 2h

### Step 1. D1 스키마 + 타입 확장 (2h)

**파일**: `infra/migrations/db-skill/0012_ai_ready_scores.sql`, `packages/types/src/ai-ready.ts`

**산출물**:
- `ai_ready_scores` 테이블
  ```sql
  CREATE TABLE ai_ready_scores (
    id TEXT PRIMARY KEY,
    skill_id TEXT NOT NULL,
    organization_id TEXT NOT NULL,
    criterion TEXT NOT NULL,         -- 6 enum values
    score REAL NOT NULL,             -- 0.0 ~ 1.0
    rationale TEXT NOT NULL,
    passed INTEGER NOT NULL,         -- 0/1
    pass_threshold REAL NOT NULL DEFAULT 0.75,
    model TEXT NOT NULL,             -- haiku | opus
    batch_id TEXT,                   -- nullable (ad-hoc re-eval)
    evaluated_at TEXT NOT NULL,      -- ISO 8601
    cost_usd REAL NOT NULL,
    FOREIGN KEY (skill_id) REFERENCES skills(skill_id)
  );
  CREATE INDEX idx_ai_ready_skill ON ai_ready_scores(skill_id, model, evaluated_at DESC);
  CREATE INDEX idx_ai_ready_batch ON ai_ready_scores(batch_id) WHERE batch_id IS NOT NULL;
  CREATE INDEX idx_ai_ready_org ON ai_ready_scores(organization_id, evaluated_at DESC);
  ```
- `ai_ready_batches` 테이블
  ```sql
  CREATE TABLE ai_ready_batches (
    batch_id TEXT PRIMARY KEY,
    organization_id TEXT NOT NULL,
    model TEXT NOT NULL,
    status TEXT NOT NULL,            -- queued|running|completed|failed
    total_skills INTEGER NOT NULL,
    completed_skills INTEGER NOT NULL DEFAULT 0,
    failed_skills INTEGER NOT NULL DEFAULT 0,
    total_cost_usd REAL NOT NULL DEFAULT 0,
    started_at TEXT NOT NULL,
    completed_at TEXT
  );
  ```
- `packages/types/src/ai-ready.ts` 확장 — `AIReadyBatchSchema` + API 요청/응답 스키마 5종 추가 (기존 AIReadyEvaluationSchema 재활용)

**검증**: Schema parse 테스트 + migration apply (local wrangler d1 execute) + 기존 F356-A 테스트 유지

### Step 2. API 엔드포인트 (2h)

**파일**: `services/svc-skill/src/routes/ai-ready.ts`, `services/svc-skill/src/ai-ready/evaluator.ts`, `services/svc-skill/src/ai-ready/repository.ts`

**엔드포인트 4종**:
1. `POST /skills/:id/ai-ready/evaluate` — 단건 동기 평가(max ~30s, 6 LLM call 직렬). 기존 D1 row 있으면 model/timestamp 기준 upsert. RBAC Analyst/Developer.
2. `POST /skills/ai-ready/batch` — 비동기 배치 트리거. body: `{ model: "haiku", organizationId: "LPON", crossCheckSampleSize?: 100 }`. 응답: `{ batchId, totalSkills, estimatedCostUsd, estimatedDurationMinutes }`. Cloudflare Queue 발송.
3. `GET /skills/:id/ai-ready/evaluations` — 이력 조회 (model/criterion filter + pagination).
4. `GET /skills/ai-ready/batches/:batchId` — 배치 상태 + 진행률.

**evaluator.ts**: CLI 로직(evaluate.ts)을 Worker 환경 버전으로 이식 — `callLlmRouterWithMeta` 사용, 비용 가드는 배치 레벨(ai_ready_batches.total_cost_usd 체크).

**repository.ts**: `insertScores(batch)`, `insertBatchHeader`, `updateBatchProgress`, `listScoresBySkill`, `getBatch` 5개 헬퍼.

**검증**: 라우트 테스트 8건 (인증 unauthorized/forbidden + 기본 동작 + batch id 생성 + 404 skill + RBAC Client 차단)

### Step 3. 전수 배치 스크립트 + Queue consumer (2h)

**파일**: `scripts/ai-ready/batch-evaluate.ts`, `scripts/ai-ready/sample-loader.ts` 확장, `services/svc-skill/src/queue/ai-ready-consumer.ts` 신규

**배치 흐름**:
1. **Trigger**: `scripts/ai-ready/batch-evaluate.ts --model haiku --cross-check 100` 실행 → production API `POST /skills/ai-ready/batch` 호출 → batchId 응답
2. **Queue fan-out**: svc-skill API 핸들러가 859 skill을 Cloudflare Queue에 1 skill = 1 message로 발송 (`ai-ready-queue`). message 메타: `{ batchId, skillId, model }`
3. **Consumer**: `ai-ready-consumer.ts`가 message 1건당 SkillPackage R2 로드 → 6 criterion LLM 호출 → `insertScores` → `updateBatchProgress`. Concurrency는 Cloudflare Queue `max_concurrency: 10` 설정.
4. **Cross-check**: Haiku 전수 완료 후 별도 Opus 100건 배치 자동 트리거 (sampling random 100/859). cross-check batchId는 parent batch 메타에 연결.
5. **CLI 완료 대기**: 스크립트는 `GET /skills/ai-ready/batches/:batchId` 폴링 (5초 간격, 최대 45분 대기).

**sample-loader 확장**: 
- 기존 fs 모드(`.decode-x/spec-containers/lpon-*`) 유지
- 신규 API 모드: svc-skill production API(`GET /skills?organization_id=LPON&limit=1000`) + R2 SkillPackage 로딩 → SpecContent 변환 (rules/runbooks/tests/contract fields 매핑)

**검증**: Dry-run 모드(10건 샘플) PASS + cost 추정 실측과 ±10% 일치 + Queue consumer unit test 3건

### Step 4. 전수 배치 실행 + KPI 리포트 (2h)

**실행**:
- Production 사용자 터미널에서 `pnpm tsx scripts/ai-ready/batch-evaluate.ts --model haiku --cross-check 100 --env production` 1회 실행
- 예상 소요: 30~40분 (Haiku 5,154 call ~34분 + Opus 600 call ~4분)
- 예상 비용: $~48 (Haiku $~20 + Opus $~28)
- **비용 가드**: $50 초과 시 자동 중단(API 핸들러 체크) + sampling 전환 제안 AskUserQuestion

**산출물**:
- `reports/ai-ready-full-2026-04-24.json` — 5,154 + 600 = 5,754 점수 (Haiku 전수 + Opus 교차검증)
- `reports/ai-ready-full-2026-04-24.md` — 분포 히스토그램(10개 bin) + 기준별 평균/중앙값/표준편차 + PASS rate + Opus vs Haiku |diff| 분포 + 위험 skill 톱10
- D1 `ai_ready_scores` production 5,154 row + `ai_ready_batches` 2 row

**KPI 판정**:
- ✅ **성공**: 전수 배치 30~40분 내 완료 + API 4종 정상 응답 + Haiku/Opus 평균 |diff| < 0.1
- ⚠️ **부분 성공**: 완료는 되었으나 Haiku/Opus |diff| ≥ 0.15 → prompt rubric v3 TODO 등록
- ❌ **실패**: 배치 45분 초과 또는 cost guard stop → sampling(100건)로 scope 축소 + 후속 Sprint 연기

## 5. Risks & Mitigation

| Risk | 확률 | 대응 |
|------|:----:|------|
| 859 skill Queue fan-out 시 Cloudflare Queue 제한 초과 | 중 | max_concurrency 10 + max_batch_size 100 설정, 필요 시 batch_size 조정 |
| Haiku 호출 5,154회 중 일부 rate limit 또는 transient fail | 중 | Queue retry 3회 + DLQ로 실패 수집, 재처리 CLI(`--retry-batch`) |
| 실행 45분 초과 (30분 목표 미달) | 중 | concurrency 10→20 증가 검토 (CF Queue 설정), 또는 sampling 모드 fallback |
| Opus 교차검증 $28 초과 | 저 | 100건 sampling 상한 + 가드 $30 |
| API 엔드포인트 Security(RBAC) 구멍 | 중 | 기존 skills evaluate 라우트 패턴 준수 + 라우트 테스트 8건 필수 |
| 859 skill SkillPackage R2 조회 시 일부 누락(TD-32 잔재) | 저 | loader에서 404 skip + 실패 리스트 리포트, 누락 < 5% 허용 |
