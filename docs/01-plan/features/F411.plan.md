# F411 Zip 내부 spec coverage 가시화 — Plan

> **Summary**: 27개 zip 업로드 전수 점검(세션 247) 결과, evidence/analysis-report UI는 zip 1건을 1 row로만 표현하여 압축 풀린 내부 chunk 단위 spec 추출 결과(API endpoint/Table/Transaction 건수)를 가시화하지 못한다. 또한 (a) `SourceProjectSummary`만 보유한 lib-only zip은 factcheck source_documentIds에 누락되고, (b) `MAX_FILE_SIZE=500KB`/`MAX_FILES=5000` 한도에 걸린 zip은 사용자에게 부분 추출이라는 사실이 미통보된다. 본 F-item은 (1) zip 내부 spec 추출 매트릭스 인라인 + (2) lib-only 라벨 + (3) 부분 추출 경고 % 3목표를 단일 Sprint(240)에 통합 구현한다.

## 1. 배경 / Problem Statement

### 1.1 발견 경로
세션 247 `/ax:session-start` 진단으로 LPON 27 zip 전수 점검 수행. 결과:

| 항목 | 실측치 |
|------|-------|
| 총 zip 업로드 | 27건 |
| `extractions.completed` 비율 | 27/27 (100%) |
| avg `process_node_count` | 2.67 |
| avg `entity_count` | 8.15 |
| avg `result_json` size | 1.4KB |
| factcheck `6e56af17` source_documentIds | 21/27 (누락 6건) |

### 1.2 누락 6건 분류
- **lib-only 4건** (`lpon-src-{contents,d7lib,extapi-mobile,extapi-hrs-server}`): chunk_cnt ≤ 1, controller/transaction/ddl 없이 `SourceProjectSummary` chunk만 존재. source-aggregator switch case가 `SourceProjectSummary` element_type을 `default: break`로 무시(`source-aggregator.ts:281`) → factcheck contributing source 등록 안 됨.
- **partial extract 2건**:
  - `companybatch.zip` (25MB): `MAX_FILE_SIZE=500KB` cap에 걸려 대부분 file skip. 사용자에게 미통지.
  - `kafka.zip` (316KB, 25 chunks): zip 자체는 정상 처리되었으나 작은 lib-only 분류로 추출 0건.

### 1.3 근본 원인 (3축)

| 축 | 위치 | 현 동작 | 문제 |
|----|------|---------|------|
| **UI 가시화** | `TriageView.tsx`, `source-upload.tsx` | zip별 1 row + Stage 2 LLM count만 표시 | AST/structured chunk 경로의 실 spec 추출량(controller/endpoint/table/transaction) 미가시 |
| **Aggregator 분류** | `source-aggregator.ts:281` | `SourceProjectSummary` skip | lib-only zip이 factcheck "contributing source"에서 제외됨 |
| **Extractor 통계 손실** | `zip-extractor.ts:54~73` | skipped/oversized는 `logger.warn`만 | chunk metadata에 추출률 정보 미저장 → UI에서 부분 추출 표시 불가 |

### 1.4 데이터 가용성 (이미 보유한 정보)
`zip-extractor.ts:139~160`이 생성하는 `SourceProjectSummary` element의 `text` JSON은 다음 stats 필드를 이미 보유:
```json
{
  "projectName": "...",
  "stats": {
    "totalFiles": 88,
    "javaFiles": 60,
    "sqlFiles": 12,
    "controllerCount": 14,
    "endpointCount": 47,
    "dataModelCount": 22,
    "transactionCount": 31,
    "ddlTableCount": 8,
    "mapperCount": 9
  }
}
```
→ UI는 별도 LLM 호출 없이 이 chunk를 fetch하여 매트릭스 렌더 가능.

**필요 추가 stats (zip-extractor 보강)**:
- `totalEntriesInZip` (Object.keys(unzipped).length)
- `skippedBinaryCount` (SKIP_PATTERNS 매칭)
- `oversizedSkippedCount` (MAX_FILE_SIZE 초과)
- `extractionRate` (extracted / totalEntriesInZip 비율)
- `cappedAtMaxFiles` (boolean — MAX_FILES 한도 도달 여부)

## 2. 목표 / Goals

### 2.1 사용자 가치
- **본부장(Executive)**: zip 한 건이 "충분히 분석되었나?" 한눈 판단. 부분 추출 zip은 "이 데이터로 의사결정해도 되나?" 위험 인지.
- **엔지니어(Engineer Workbench)**: 어떤 zip의 어떤 component(controller/table/transaction)가 부족한지 즉시 식별 → 추가 업로드 또는 재업로드 결정.

### 2.2 KPI / Success Criteria

| KPI | 현재 | 목표 |
|-----|------|------|
| zip 내부 spec 매트릭스 가시화 | 0% (UI 미존재) | 100% (모든 parsed zip에 endpoint/table/transaction count 인라인 표시) |
| lib-only zip 라벨 정확도 | 0% (미분류) | ≥ 95% (controller=0 + dataModel=0 + transaction=0 + mapperCount=0 + summaryOnly=true) |
| 부분 추출 경고 노출 | 0건 | 100% (extractionRate < 0.95 또는 cappedAtMaxFiles=true 또는 oversizedSkippedCount > 0 zip에 모두 노출) |
| factcheck source_documentIds lib-only 인지 | lib-only 누락 | lib-only를 별도 marker(`includes lib-only count`)로 응답 메타에 노출 |
| **E2E 스모크** (UX F-item Must, SPEC §4 #6) | 0건 | 1건 (source-upload + analysis-report에서 27 zip 중 lib-only 1건 + partial 1건 정상 표기 검증) |

### 2.3 Out-of-Scope (명시적 제외)
- Stage 2 LLM 재호출 (avg process_node_count 2.67 자체 개선) — 별도 REQ
- `MAX_FILE_SIZE`/`MAX_FILES` 상수 자체 상향 — 본 F-item은 "현재 한도 + 가시화"만 (별도 TD 등록 후보)
- factcheck rerun 전수 (lib-only 포함 재집계) — 본 F-item은 metadata 노출만, 재실행은 사용자 액션

## 4. 데이터 흐름 (현재 → 목표)

### 4.1 현재 (As-Is)
```
zip 업로드
  → svc-ingestion zip-extractor (skipped/oversized는 log only)
    → document_chunks 저장 (SourceProjectSummary stats 9 필드)
      → svc-extraction source-aggregator (SourceProjectSummary skip)
        → factcheck source_documentIds (lib-only 누락)

UI:
  - source-upload.tsx: chunk classification 4종 카운트만
  - TriageView.tsx: zip별 ruleCount/relCount/entCount/procCount만
```

### 4.2 목표 (To-Be)
```
zip 업로드
  → svc-ingestion zip-extractor
    [+] skipped/oversized/totalEntries/extractionRate/cappedAtMaxFiles 카운터
    → document_chunks 저장 (SourceProjectSummary stats 14 필드)
      → svc-extraction source-aggregator
        [+] SourceProjectSummary 처리 → libOnlyDocumentIds 응답
        → factcheck source_documentIds
          [+] lib-only marker 동반 응답

UI:
  - source-upload.tsx: chunk count + endpoint/tx/lib-only badge + "부분 추출 X%"
  - TriageView.tsx: row expand → spec 매트릭스 인라인 (endpoint/dataModel/tx/ddl/mapper)
```

### 4.3 lib-only 판정 규칙 (확정)
```
isLibOnly = (
  chunks.length === 1 &&
  chunks[0].element_type === "SourceProjectSummary" &&
  stats.controllerCount === 0 &&
  stats.dataModelCount === 0 &&
  stats.transactionCount === 0 &&
  stats.mapperCount === 0
)
```
또는 chunks.length > 1이지만 위 4개 count 모두 0인 경우(예: utility class만 있는 zip).

### 4.4 부분 추출 경고 규칙 (확정)
```
isPartialExtraction = (
  extractionRate < 0.95 ||
  cappedAtMaxFiles === true ||
  oversizedSkippedCount > 0
)

severity:
  - HIGH: extractionRate < 0.5 || cappedAtMaxFiles
  - MEDIUM: 0.5 ≤ rate < 0.8
  - LOW: 0.8 ≤ rate < 0.95 || only oversized skips
```

## 5. 구현 단계 (Steps)

### Step 1 — Backend 통계 보강 (1.5h)
**Files**: `services/svc-ingestion/src/parsing/zip-extractor.ts`

- `extractSourceFiles` 내부에 카운터 4개 추가 (skippedBinary, oversizedSkipped, totalEntries, cappedAtMaxFiles)
- return type에 `ExtractionStats` 추가 (별도 return tuple 또는 export)
- `parseSourceProject`의 `SourceProjectSummary.stats` JSON에 5개 신규 필드 추가
- Unit test: `parseSourceProject.test.ts` (없으면 신설) — mock files 입력으로 stats 14 필드 모두 정확히 카운트되는지 검증
- Test: oversized file 1건 + binary file 1건 + 정상 5건 → extractionRate=0.71, oversizedSkippedCount=1

### Step 2 — Aggregator lib-only 처리 (1.5h)
**Files**: `services/svc-extraction/src/factcheck/source-aggregator.ts`

- `SOURCE_CLASSIFICATIONS` set 그대로 유지 (`source_project` 이미 포함)
- 메인 loop에 `SourceProjectSummary` element_type 처리 case 추가:
  - parse된 stats가 lib-only 규칙 매칭이면 `libOnlyDocumentIds.push(doc.document_id)`
  - matched zip의 source_file/projectName을 `SourceSpec.libOnlyProjects` 배열에 추가 (신규 필드, optional)
- `SourceSpec` type 확장 (`packages/types/src/factcheck.ts` 또는 source-aggregator/types.ts) — `libOnlyProjects?: Array<{ documentId: string; projectName: string; reason: string }>`
- Unit test: lib-only zip 1건 + 정상 zip 1건 → libOnlyProjects.length===1 검증

### Step 3 — Triage API 응답 확장 (1.5h)
**Files**: `services/svc-extraction/src/routes/analysis.ts`, `packages/types/src/analysis.ts`

- `TriageDocumentSchema`에 optional 3 필드 추가:
  - `chunkSummary?: ZipChunkSummary` (endpoint/dataModel/transaction/ddl/mapper count + extraction stats)
  - `isLibOnly?: boolean`
  - `partialExtraction?: { rate: number; severity: 'HIGH'|'MEDIUM'|'LOW'; reasons: string[] } | null`
- `handleGetTriage`에서 zip 문서(`file_type === 'zip'` 또는 `original_name.endsWith('.zip')`)에 한해 추가 chunk fetch → `SourceProjectSummary` chunk parse → 위 필드 채움
- 비-zip 문서는 위 필드 undefined (UI에서 fallback 렌더)
- N+1 쿼리 회피: 한 번에 모든 zip의 SourceProjectSummary chunk만 batch fetch (D1 `WHERE element_type='SourceProjectSummary' AND document_id IN (?,?,...)`)
- Integration test: 27 zip mock → 응답 검증, lib-only 4건 식별

### Step 4 — UI 통합 + E2E (1.5~2h)
**Files**: `apps/app-web/src/components/analysis-report/TriageView.tsx`, `apps/app-web/src/pages/source-upload.tsx`, `apps/app-web/e2e/zip-coverage.spec.ts` (신설)

- **TriageView**:
  - `TriageRow`에 expand chevron 추가 (zip 문서만)
  - expand 시 sub-row에 spec 매트릭스 표 (endpoint X / table Y / tx Z / mapper W)
  - lib-only zip은 row에 `<Badge variant="secondary">라이브러리</Badge>`
  - partial extraction은 severity별 색상 badge + tooltip(추출률 % + 사유)
- **source-upload.tsx**:
  - 기존 `chunkCounts` state에 `summary` 추가 (controllerCount/endpointCount/dataModelCount/transactionCount/mapperCount)
  - chunk count 영역에 spec 매트릭스 텍스트 추가 ("API 47 / Table 8 / Tx 31")
  - lib-only/partial badge 추가
- **E2E** (`apps/app-web/e2e/zip-coverage.spec.ts`):
  - vite dev + `VITE_DEMO_MODE=1` 모드에서 mock zip 3종 (정상 / lib-only / partial) 데이터 fixture 주입
  - source-upload 페이지 — 3 zip 모두 표시 + 정상 zip endpoint count > 0 / lib-only badge / partial 경고 % 검증
  - analysis-report TriageView — 정상 zip expand → 매트릭스 표시 검증
