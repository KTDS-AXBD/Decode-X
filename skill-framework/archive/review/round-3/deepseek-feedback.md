## 기술 검토 의견: Skill Framework PRD v2

### 1. 기술적 실현 가능성

**긍정적 측면:**
- Markdown + YAML frontmatter는 적절한 기술 선택. 경량이며 Claude Code와 자연스럽게 통합 가능
- Git 기반 배포는 기존 인프라 활용 측면에서 실현 가능성 높음
- Bash/Node.js 검증 도구는 복잡도 대비 적절한 선택

**기술적 제약 및 병목:**
- **실시간 동기화의 복잡성**: `watchdog` 기반 파일 시스템 감시는 이론적으로 가능하나, 실제 구현 시 다음과 같은 문제 발생 가능:
  - WSL/Mac 간 파일 시스템 이벤트 처리 차이
  - 대규모 파일(210+ 스킬) 변경 감지 시 성능 저하
  - 네트워크 드라이브나 공유 폴더에서의 신뢰성 문제
- **의존성 관리의 숨겨진 복잡도**:
  - 순환 의존성 탐지는 그래프 알고리즘 필요(O(n²) 이상 복잡도 가능)
  - 버전 호환성 검증은 실제로 semantic versioning 파싱 및 비교 로직 필요
  - 스킬 간 암시적 의존성(import, require)까지 추적하려면 정적 분석 도구 필요
- **사용량 추적의 정확성 문제**:
  - `PreToolUse` 훅만으로는 "실제 사용"과 "실패한 호출" 구분 어려움
  - `${CLAUDE_PLUGIN_DATA}` 저장소의 크기 제한 및 정리 정책 미정의

### 2. 아키텍처 적합성

**현재 설계의 문제점:**
- **하이브리드 아키텍처의 모호성**: "Core Registry + Distributed Storage"는 개념적으로 좋으나:
  - 데이터 일관성 보장 메커니즘이 부족(최종적 일관성만 언급)
  - 분산 저장소 간 충돌 해결 전략(conflict resolution) 미정의
  - 장애 시 fallback 조회가 실제로 어떻게 작동하는지 구체적이지 않음

**대안 제안:**
- **계층적 캐싱 아키텍처** 고려:
  ```
  Local Cache (각 클라이언트) ←→ Central Registry (메타데이터만) ←→ Git Storage (실제 파일)
  ```
  - 변경 시 Git push → Registry webhook → 클라이언트 polling/websocket
  - 오프라인 지원을 위한 로컬 캐시 TTL 설정
- **이벤트 소싱 패턴** 적용 검토:
  - 모든 스킬 변경을 이벤트로 기록
  - 클라이언트는 최신 이벤트만 동기화하여 증분 업데이트

**확장성 관점:**
- 단일 개발자(1명) 환경에서는 과도한 아키텍처일 수 있음
- 하지만 팀 확장(BD팀 전체)을 고려하면 적절한 설계
- 문제는 **점진적 도입 경로**가 명확하지 않음

### 3. 구현 복잡도 (과소평가된 요소)

**숨겨진 복잡도:**
1. **중복 스킬 검출의 AI/ML적 접근 필요성**:
   - "메타데이터·핵심 기능 비교"는 단순 문자열 비교로 불충분
   - 실제 기능 유사성 판단을 위해:
     - 코드 시맨틱 분석(abstract syntax tree 비교)
     - 자연어 처리(description 유사도)
     - 실행 패턴 분석(필요한 입력/출력 비교)

2. **동시성 및 락 관리**:
   - 여러 팀원이 동시에 같은 스킬 수정 시 충돌
   - Git merge conflict는 개발자에게 익숙하지만, 자동화된 시스템에서는 복잡
   - 카탈로그 인덱스 빌드 중 변경 발생 시 일관성 문제

3. **크로스플랫폼 호환성**:
   - WSL(Windows)와 Mac 간 경로 처리 차이(`~/.claude/` vs `/Users/...`)
   - 파일 권한 모델 차이(특히 실행 가능 스크립트)
   - 라인 엔딩(CRLF vs LF)으로 인한 스크립트 실패 가능성

4. **점진적 마이그레이션 부담**:
   - 210+ 기존 스킬을 새 체계로 전환하는 작업량 과소평가
   - 하위 호환성 유지 필요(기존 프로젝트에서 구식 스킬 참조 중단 방지)

### 4. 기술 리스크 (우려 순위)

**상위 리스크:**
1. **데이터 일관성 파괴** (가장 치명적):
   - 분산 저장소와 중앙 메타데이터 불일치로 인한 "스킬 찾을 수 없음" 오류
   - 의존성 그래프 깨짐으로 인한 연쇄 실패

2. **성능 저하로 인한 사용자 거부**:
   - 카탈로그 조회 시 2초 이상 지연 발생 시 사용 포기
   - 파일 시스템 감시가 CPU/메모리 과다 사용 가능

3. **유지보수 부담 집중**:
   - 모든 기능이 Sinclair 1명에게 의존하는 Single Point of Failure
   - 복잡한 아키텍처의 디버깅 난이도 높음

4. **Claude Code 플러그인 시스템 변경 리스크**:
   - Anthropic이 Plugin API 변경 시 대응 필요
   - 사내망 제약으로 인한 외부 플러그인 업데이트 지연
