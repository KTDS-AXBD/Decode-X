> 생성일: 2026-03-08

## 핵심 평가 — 소스코드 역공학은 가능한가?
> 전자식 온누리상품권 플랫폼의 소스코드·문서에서 도메인 지식을 추출할 수 있는가

**프로젝트 핵심 질문:**
전자식 온누리상품권 플랫폼의 소스코드와 운영 문서(규정, 매뉴얼, 사용자 가이드 등)를 AI로 분석하여, 플랫폼의 비즈니스 로직과 운영 정책을 체계적으로 추출·자산화할 수 있는가? 소스코드 API와 문서 간 갭(Gap)은 어디에 있으며, 어떻게 보완할 수 있는가?

## 종합 판정
> 핵심 평가 최종 결론

**현재 수준:** 
전자식 온누리상품권 플랫폼의 88건 문서에서 848건 정책, 7,332건 용어, 859건 Skill을 추출했어요. 도메인 전환(퇴직연금 → 온누리상품권) 시 파이프라인 재설정 없이 즉시 적용 가능함을 검증했고, 이는 AI Foundry의 도메인 비의존(domain-agnostic) 아키텍처가 유효함을 입증해요.

**차별점:** 
FactCheck 분석에서 소스코드 1,128개 API 대비 문서 커버리지가 8.7%에 불과한 점이 발견되었어요. 이는 SI 프로젝트의 구조적 문제(문서 미갱신, 암묵지 소스코드 잠금)를 정량적으로 보여주며, AI Foundry가 이 갭을 메울 수 있는 도구임을 시사해요.

**다음 단계:** 
Wave 2 (Archive 127건) 투입, 도메인 코드 자동 감지(POL-LPON-*), Neo4j backfill 완료, 그리고 소스코드 직접 분석(AST)으로 API 커버리지를 높이는 것이 우선이에요.

## 품질 평가
> 온누리상품권 도메인 추출 성과 및 한계 분석

### 잘 되는 것 — 추출 사례
- 상품권 발행·충전·사용·환불 전 과정의 업무 규칙을 조건-기준-결과 트리플로 자동 추출 (848건)
- 규정집, 매뉴얼, FAQ 등 다양한 문서 유형에서 일관된 정책 구조화 — 파싱 성공률 96.6% (85/88)
- 도메인 용어 밀도가 퇴직연금 대비 2.5배 — 온누리상품권 특화 용어 사전으로 즉시 활용 가능
- 벌크 승인 333건 운영 — 대량 정책 검토·승인 워크플로 실용성 검증

### 한계점
- 소스코드 ↔ 문서 커버리지 8.7% — 소스에 존재하는 API의 91.3%가 문서에 미반영
- 정책 코드 POL-PENSION-* 하드코딩 — 온누리상품권 도메인 코드가 아닌 퇴직연금 코드로 생성됨
- PPTX 대용량 파일 1건 파싱 실패 (524 timeout) — Cloudflare 30초 제한
- Neo4j 그래프 backfill 3,752건 대기 — 온톨로지 시각화·검색 미완
