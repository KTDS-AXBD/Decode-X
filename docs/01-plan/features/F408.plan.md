## 1. Overview

### 1.1 Purpose

`services/svc-skill/src/ai-ready/evaluator.ts`의 `loadSpecContent()`가 R2 `spec-containers/{org}/{skillId}/manifest.json` + 하위 파일 6종(originalRules/emptySlotRules/runbooks/tests/contractYaml/provenanceYaml)을 기대한다. 그러나 production R2 `ai-foundry-skill-packages` 버킷은 Sprint 228 F397 packaging 결과인 `skill-packages/{skillId}.skill.json` 단일 번들만 존재한다 (`wrangler r2 object list spec-containers/` 결과 0건 확인, S238 계속 2 Master smoke).

본 F-item은 **evaluator의 데이터 로딩 계층을 SkillPackage 직접 파싱으로 전환**하여 LPON 859 production skill 전수에 대해 즉시 채점 가능한 상태를 만든다. 기존 spec-container 경로는 `loadSpecContentLegacy()`로 보존하여 회귀 안전성을 확보한다.

### 1.2 Background

- **선행 해소 (Sprint 238 F356-B)**: 코드 100% 완결 (typecheck clean + 402/402 tests PASS + Match 96% + Migration SUCCESS), Production CI/CD Deploy SUCCESS
- **차단 발견 (Sprint 238 계속 2)**: Master 독립 production smoke 3차 호출 시 `POST /skills/{lpon-charge-uuid}/ai-ready/evaluate` → HTTP 404 `{code:"NOT_FOUND", message:"spec-container '...' not found"}` 응답. Phase 1 (D1 skill 조회) 정상 통과, Phase 2 (R2 manifest 로드) 실패
- **autopilot 7회차 sprint-smoke-mismatch 패턴 (S215/S219/S220/S228/S230/S232/S238)**: Match 96% + CI green + typecheck clean에도 production R2 layout 미스매치 자체 탐지 불능 → Master 독립 실측만이 검증 가능
- **TD-47 (P1) 등록 (S238 계속 2)**: 2안 중 선택 — (A) `migrate-spec-containers-to-r2.ts` 7개만 (852 부재, 데모용) vs (B) evaluator를 `.skill.json` 직접 파싱 전환. 사용자 결정: **(B) 채택**

### 1.3 Scale & Risk 산정

- **변경 코드 라인**: ~150 lines 추가 (adapter + evaluator 분기) + ~50 lines 테스트
- **production 영향 범위**: svc-skill 배포 1회 (`wrangler deploy --env production`)
- **score baseline drift 위험**: spec-container vs skill-package 데이터 소스 차이로 6기준 점수가 변동 가능
  - **comment_doc_alignment** (runbooks/*.md 필요): SkillPackage에 runbook 부재 → 자동 0~0.3 가능성
  - **io_structure** (tests/contract/*.yaml 필요): SkillPackage에 contract 부재. 단 `technicalSpec.apis`로 대체 가능성 → 0.5~0.7
  - **exception_handling** (rules + tests): SkillPackage에 tests 부재 → 0.4~0.6
  - **source_consistency** (originalRules vs provenance.businessRules): SkillPackage policies 자체가 provenance와 동일 출처 → 0.85~0.95 유지 예상
  - **comment_alignment + srp + testability**: 부분 영향
- **|Δscore| ≤ 0.05 DoD 재정의 필요**: 데이터 소스 자체가 다르므로 동일 baseline은 비현실적. 대신 **production-only baseline 신규 등록 + Phase 2 운영 실행 시 사용**

## 3. Tasks (4 Steps)

### Step 1 — SpecContent Adapter (1h)

**파일**: `services/svc-skill/src/ai-ready/spec-content-adapter.ts` (신규)

**책임**:
- SkillPackage Zod schema(`@ai-foundry/types`)에서 `policies / provenance / technicalSpec / metadata`를 읽어 SpecContent 형식으로 정규화
- 누락 가능 필드(runbooks/tests)는 빈 string `""` 또는 빈 배열 `[]`로 자리 비우기 (rubric 평가에서 LLM이 "부재" 판정하도록 명시 메시지 포함)

**핵심 함수**:
```ts
export function skillPackageToSpecContent(pkg: SkillPackage): { specContent: SpecContent; skillName: string } {
  const skillName = `${pkg.metadata.domain}-${pkg.metadata.subdomain ?? "default"}`;
  return {
    skillName,
    specContent: {
      rules: [policiesToOriginalRules(pkg.policies)],
      originalRules: [policiesToOriginalRules(pkg.policies)],
      emptySlotRules: [],  // skill-package에는 ES 분리 정보 없음
      runbooks: [],          // skill-package에는 runbook 정보 없음
      tests: [],             // skill-package에는 test 정보 없음
      contractYaml: technicalSpecToContractYaml(pkg.technicalSpec),
      provenanceYaml: provenanceToYaml(pkg.provenance),
    },
  };
}
```

**markdown rules format** (`policiesToOriginalRules`):
```markdown
| ID       | condition          | criteria       | outcome        | exception      |
|----------|--------------------|----------------|----------------|----------------|
| BL-001   | {policy.condition} | {criteria}     | {outcome}      | —              |
| BL-002   | ...                | ...            | ...            | ...            |
```
- `code` (POL-{DOMAIN}-{TYPE}-{SEQ})에서 SEQ 부분만 추출하여 BL-{SEQ} 형식 사용 (rubric "ID 문자열 일치" 호환)
- `exception`은 SkillPackage에 부재 → 일관되게 `—` (em dash)로 표기 (rubric 0.9+ "exception 열 완결성 필수" 충족)

### Step 2 — evaluator.ts 교체 (1h)

**파일**: `services/svc-skill/src/ai-ready/evaluator.ts` (수정)

**변경**:
1. 새 함수 `loadSpecContentFromSkillPackage()` 추가
   ```ts
   export async function loadSpecContentFromSkillPackage(
     env: Env,
     skillId: string,
   ): Promise<{ specContent: SpecContent; skillName: string } | null> {
     const key = `skill-packages/${skillId}.skill.json`;
     const obj = await env.R2_SKILL_PACKAGES.get(key);
     if (!obj) {
       logger.warn("skill-package not found in R2", { skillId, key });
       return null;
     }
     const raw = await obj.text();
     const pkg = SkillPackageSchema.parse(JSON.parse(raw));
     return skillPackageToSpecContent(pkg);
   }
   ```
2. 기존 `loadSpecContent()` 수정 — fallback 시퀀스
   ```ts
   // 1) 우선 skill-packages 시도 (production)
   const fromPkg = await loadSpecContentFromSkillPackage(env, skillId);
   if (fromPkg) return fromPkg;
   // 2) 없으면 spec-containers fallback (legacy/dev)
   return loadSpecContentLegacy(env, skillId, organizationId);
   ```
3. 기존 R2 manifest 로딩 로직을 `loadSpecContentLegacy()`로 rename + export

## 5. Risk & Mitigation

### R1: skill-package 데이터로는 6기준 평가가 spec-container 대비 현저히 낮게 나옴 (특히 comment_doc_alignment / io_structure / exception_handling)

**Likelihood**: HIGH (runbooks/tests/emptySlot 자체가 부재)

**Mitigation**:
- (a) 본 Sprint에서는 baseline 차이를 정량화만 (rubric 재조정 NO) — Sprint 240 진입 전 판단 자료 확보가 우선
- (b) 차이가 크면 Sprint 240에서 rubric에 "데이터 소스별 가중치" 추가 또는 (c) skill-package에 runbook/test 필드 보강 (별도 F-item)
- (c) DoD를 |Δscore| ≤ 0.05 절대값에서 "기능 동작 PASS + 차이 정량화"로 재정의 (본 Plan에서 명시)

### R2: SkillPackageSchema parse 실패 (production .skill.json이 schema 진화에 뒤처짐)

**Likelihood**: MEDIUM

**Mitigation**:
- adapter에서 `safeParse()` 사용 + 실패 시 logger.error + null 반환 (graceful)
- 실 production R2의 .skill.json 1건 sample 사전 fetch + parse 검증 후 코드 작성
- schema 진화 발생 시 별도 migration F-item (TD 등록)

### R3: evaluator.ts가 svc-skill production 배포 시점에 in-flight 요청과 충돌

**Likelihood**: LOW (현재 batch 미가동)

**Mitigation**:
- 변경이 backward-compatible (legacy fallback 유지) — 기존 spec-container 경로 hit 시 동작 변경 없음
- production 배포 직후 health check 1회 + smoke 1회 즉시 수행
